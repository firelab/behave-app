#+TITLE: Behave7 Startup Optimization Plan
#+DATE: 2026-07-07
#+AUTHOR: RJ Sheperd (w/ Claude)

* Overview

Behave7 startup has two surfaces, and both are almost entirely *serial* today:

1. *Backend / desktop (JVM + JCEF)* — =behave.core/-main= → config → logging →
   DataScript/SQLite restore → *then* CEF/Chromium init. Nothing overlaps.
2. *Frontend (every launch, desktop and web)* — =behave-min.js= (343 KB) →
   WASM instantiate (454 KB) → =app.js= (2.7 MB) download+parse →
   =behave.client/init= → fetch =layout.msgpack= (1.4 MB) + =/api/sync=
   (full user-datom dump) → synchronous =d/transact= of both → first real paint.

The single biggest structural problem: *five sequential phases* on the frontend
critical path where at least three could run concurrently, plus zero HTTP
compression and no client-side caching of the two boot payloads.

* Current Startup Sequence (as measured from code)

** Backend (=projects/behave/src/clj/behave/core.clj:115=)
1. =init-config!= / =enrich-config!= — trivial (~400 B EDN).
2. =show-loader!= — Swing splash appears early (good).
3. =start-logging!= + =log-system-start!= — logs *every* JVM system property
   and full config synchronously (=core.clj:14-18=).
4. =server/init-db!= → =datom-store.main/connect!=
   (=bases/datom_store/src/datom_store/main.clj:66=):
   - =d/restore-conn= — deserializes the *entire* SQLite-backed DataScript DB
     into memory. Scales with user's worksheet history.
   - =build-tx-index!= (=main.clj:35=) — full =d/datoms :eavt= scan, split,
     filter, group-by tx. A second O(all-datoms) pass at every boot.
5. =create-cef-app!= — CEF/Chromium native init on the Swing EDT. Only
   *scheduled after* DB init completes, so the two slowest backend steps are
   serialized.
6. First page request: =reset-vms-version!= (=views.clj:145=) slurps the 1.4 MB
   =layout.msgpack= and MD5s it synchronously before the first HTML byte.

** Frontend (=projects/behave/src/clj/behave/views.clj:187=, =resources/onload.js=)
1. HTML → =/js/behave-min.js= (Emscripten glue).
2. =createModule()= fetches + instantiates =behave-min.wasm=.
3. *Only then* =onload.js= appends =<script src=/cljs/app-<hash>.js>= — the
   2.7 MB bundle cannot even begin downloading until WASM is done. (Ordering
   exists because every =behave.lib.*= namespace reads =js/Module= at
   namespace-load time, e.g. =behave/lib/enums.cljs:8=.)
4. =behave.client/init= (=client.cljs:148=) → fires =load-vms!=
   (=/layout.msgpack=, 1.4 MB) and =load-store!= (=/api/sync=, full dump) —
   these two *are* parallel with each other, but start only after ~3.5 MB of
   JS/WASM has serially loaded and parsed.
5. =app-shell= renders =Loading...= until *both* =vms-loaded?= and
   =sync-loaded?= — each requires msgpack unpack + a synchronous main-thread
   =d/transact= of the full datom set.
6. =katex.min.js= (264 KB) + =bodymovin.js= (531 KB) load eagerly on every page.

Everything is served *uncompressed* (no gzip middleware in
=handlers.clj:180-196=) and both boot payloads are re-fetched on every launch
(no service worker / IndexedDB / localStorage caching; only =behave-settings=
uses localStorage).

Deployed the JDK 21 build via =scripts/test-windows.sh= (v7.1.6 test zip,
unsigned). Box: 2-core VM, Xeon E5-2699 v4 @ 2.2GHz. App runs, JCEF window
shows, =/api/sync= served. Numbers:

| Metric                         | Cold (fresh files) | Warm relaunch |
|--------------------------------+--------------------+---------------|
| scheduling CEF app (JVM start) | 33.3s              | 33.8s         |

| datom-store restore (async)    | 11.4s              | 586ms         |
| browser window shown           | 45.8s              | 35.2s         |

Caveats and analysis:
- The =schtasks=-based launcher used by the test script runs at *below-normal
  priority*, inflating CPU-bound phases ~3-4x. Launching =java -cp
  app/behave7.jar clojure.main -e "(require 'behave.core)"= directly on the
  box: *~9s wall, ~17s CPU* (stable across runs; Defender CPU ~0 — not AV).
  Real double-click launches should see roughly 9-12s to splash, not 33s.
- *App class loading is THE desktop bottleneck on weak Windows hardware*:
  17 CPU-seconds (vs 2.3 on an M-series Mac). The async DB/hash work
  (Phase 2) is confirmed off the critical path (restore hides behind CEF).
- Biggest remaining lever, in order:
  1. *Defer the heavy namespace tree out of =behave.core='s ns load* —
     =behave.handlers= drags in datascript/core.async/schemas before =-main=
     runs, so the splash can't show for ~9s. =requiring-resolve= the handler
     stack inside =start-cef!= after =show-loader!= (like jcef already is)
     → splash at ~1-2s, rest loads while CEF boots.
  2. AppCDS (~2x on those 17 CPU-sec) — blocked on Conveyor cross-build,
     see 2.5.

**** DONE Splash-first entry (item 1) — =behave.Launcher=, implemented 2026-07-07

Final form is a *pure-Java* entry class,
=projects/behave/src/java/behave/Launcher.java= (compiled via a new
=b/javac= step in =packaging.clj=; Conveyor =jvm.gui= and uberjar
=:main-ns= point at =behave.Launcher=). It shows the splash (same layout
as =jcef.loading=) before any Clojure class loads, then boots Clojure via
=clojure.java.api= and hands the frame to =behave.core/start-cef!=.
Server mode delegates to =behave.core/-main= unchanged.

An intermediate Clojure shim (=behave.main=, since deleted) got the splash
to 7.3s — but even an empty Clojure namespace pays ~5-6s of =clojure.core=
class-init on the slow box. Java skips that entirely. Keep =Launcher.java=
free of Clojure references before the splash: the first =Clojure.var()=
call triggers the =clojure.core= load.

Measured on the Windows test box (direct launcher runs, same session):
| Milestone             | Original | Clojure shim | Java Launcher |
|-----------------------+----------+--------------+---------------|
| entry code running    | —        | 6.7s         | *0.25s*       |
| splash visible        | ~30s     | 7.3s         | *0.7s*        |
| app namespaces loaded | ~30s     | ~30s         | ~30s (behind splash) |
(Mac: launcher entered 76ms vs 881ms for the Clojure entry.)

Also assessed: the JVM's =-splash:= flag (~0.2s, zero code) — not viable:
it's implemented by the =java= launcher binary, and Conveyor's custom
launcher has no documented support for it. Remaining lever for the ~30s
tree load itself: AppCDS (2x, blocked on Conveyor cross-build, see 2.5).

**** DONE Animated progress bar + startup checkpoints — implemented 2026-07-07

The Launcher splash now has a =JProgressBar= that eases toward each
checkpoint and creeps slowly between them (capped ~8% past target), so it
never looks frozen during long stages. The Launcher drives the namespace
load in *stages* — each =require= is a checkpoint with a user-facing
message — and hands a =BiConsumer<Integer,String>= to
=behave.core/start-cef!= for the tail-end checkpoints
(=Launcher.setProgress= is thread-safe, EDT-dispatched).

Checkpoint set (percent → stage → measured wall-clock on the test box):
| %   | Stage / message          | Fired by            | Box timing |
|-----+--------------------------+---------------------+------------|
| 2   | splash shown             | Launcher            | 0.8s       |
| 20  | Clojure runtime          | Launcher            | 8.2s       |
| 42  | core libraries (async)   | Launcher            | 16.3s      |
| 62  | database engine (datascript) | Launcher        | 24.4s      |

| 90  | starting browser         | start-cef!          | ~33.5s     |
| 100 | browser window shown     | start-cef! on-shown | 38.7s      |

Percentages are calibrated to the box's wall-clock fractions. To
re-calibrate after the load profile changes: run =bin/Behave7.exe= from a
terminal and read the =[TIMING] stage <ns> loaded= stdout lines.
Intermediate stage requires are best-effort (a renamed ns logs a warning
and falls through; only the final =behave.core= require is load-bearing).

The user-facing messages are a grab-bag of outdoors phrases (fishing/
hunting/hiking/camping — "Pitching the tent...", "Casting the line...",
"Brewing the cowboy coffee..."), one drawn at random per stage per launch
(=grab= in =Launcher.java=; =rand-nth= for the two =start-cef!=
checkpoints). The =[TIMING]= stdout lines record both the namespace and
the drawn phrase so logs stay diagnosable.

- =file-utils/os-path= turned =C:\Users\...= into drive-relative =C:Users\...=

* Phase 1 — Frontend Critical Path [highest impact]

** 1.1 Parallelize WASM instantiate and app.js download [S effort, high impact]
The bundle only needs =window.Module= set *before it executes*, not before it
downloads.
- Add =<link rel="preload" as="script" href="/cljs/app-<hash>.js">= in
  =head-meta-css= (=views.clj:57=), and same for =behave-min.wasm=
  (=as="fetch" type="application/wasm" crossorigin=).
- Or restructure =onload.js=: fetch the bundle text in parallel
  (=Promise.all([createModule(), fetch(appJs)])=) and inject/execute it only
  after =Module= is set. Preload is simpler and sufficient.
- Longer-term (optional): make =behave.lib.*= resolve =js/Module= lazily
  (accessor fn instead of top-level def) so script order stops mattering.

** 1.2 Start boot data fetches before app.js parses [S effort, high impact]
=layout.msgpack= and =/api/sync= are known at HTML-render time.
- Emit =<link rel="preload" as="fetch" href="/layout.msgpack?v=...">= (exact
  URL already computed server-side) and =/api/sync= in the page head, or kick
  off both =fetch()= calls in the inline =cljs-init= script and stash the
  promises on =window= for =load-vms!= / =load-store!= to consume.
- Result: data downloads overlap the WASM+JS phase instead of following it.

** 1.3 Compress everything [S effort, high impact for web mode]
- Precompress at build time (=bb build-js= step): gzip/brotli =app.js=,
  =layout.msgpack=, =behave-min.js/.wasm=, css. Serve =.gz=/=.br= with
  =Content-Encoding= via middleware (e.g. =ring.middleware.gzip= or
  bsless/ring-gzip), or hand-rolled resource lookup in =handler-stack=.
- Expected: app.js 2.7 MB → ~600-700 KB; layout.msgpack shrinks substantially.
- Desktop (CEF, local scheme handler) benefit is smaller but msgpack unpack
  time is unaffected — this is a wire-size win for web/server mode primarily.

** 1.4 Cache boot payloads client-side [M effort, high impact on relaunch]
Both payloads are already content-versioned (=?v=<md5>= for layout; fingerprint
for app.js) — exploit that:
- Add =Cache-Control: public, max-age=31536000, immutable= for fingerprinted
  =/cljs/app-*.js= and versioned =/layout.msgpack?v=...= responses;
  =no-cache= for HTML. Today no cache headers are set at all, and the CEF
  =custom-request-handler= path should be checked to honor them (CEF cache-path
  is already configured in =start-cef!=).
- Optional deeper win: persist the *unpacked* VMS DataScript DB (serialized via
  =dt/write-transit-str= or datoms array) in IndexedDB keyed by =vms-version=;
  skip download *and* re-transact when version matches. Same idea for
  =/api/sync= is harder (data changes every session) — skip it.

** 1.5 Defer katex + bodymovin [S effort, small-medium impact]
- Add =defer= to both script tags, or lazy-load on first use (katex → help
  pages with math; bodymovin → loading animations). ~800 KB off the boot path.

** 1.6 Skeleton first paint [S effort, perceived-speed]
- =app-shell= can render the chrome (toolbar, sidebar skeleton) immediately and
  gate only the content area on =vms-loaded?/sync-loaded?= instead of a bare
  ~Loading...~ =h3=. Pure perceived-latency win while 1.1-1.4 land.

** 1.7 (Later) Code-splitting =:modules= [L effort, medium impact]
Split =app.js= — e.g. wizard/results/print/tools as separate closure modules,
=cljsjs/vega*= (~2 MB source) only loads with results/graphs. Real work under
figwheel-main + =:advanced=; do only if 1.1-1.5 aren't sufficient.

* Phase 2 — Backend / Desktop JVM Path

** DONE 2.1 Overlap CEF init with DB init [S-M effort, high impact on desktop cold start]
In =start-cef!= (=core.clj:70=): CEF =.build= (Chromium process spawn) and
=init-db!= (SQLite restore) are independent — run =init-db!= on a =future= and
call =create-cef-app!= immediately. Gate only the request handlers on a
db-ready promise (deref in =/api/sync=, =/api/init=, =save/open= handlers).
The browser needs ~1-2 s to boot and load the JS anyway — the DB restore hides
inside that window for free.

/Implemented 2026-07-07:/ =server/init-db-async!= runs =init-db!= on a future
(both CEF and server paths); =behave.store= exposes a =db-ready?= promise
delivered by =connect!=; =wrap-await-db= middleware in =behave.handlers=
blocks only =/api/init|sync|save|open= until the conn is ready (30 s timeout).

** 2.2 Defer or drop =build-tx-index!= at boot [S effort, medium impact, grows with DB]
=datom_store/main.clj:35= does a full =:eavt= scan only to serve
=latest-datoms= incremental sync. Options (pick one):
- Build lazily on first =latest-datoms= call (delay).
- Build on a background thread after =connect!= returns.
- Track only =max-tx= at boot; clients doing a full =/api/sync= GET don't need
  the index at all.

** DONE 2.3 Trim first-request work [S effort, small-medium impact]
- =reset-vms-version!= (=views.clj:145=): compute the layout.msgpack MD5 at
  *build time* (write alongside =version.edn=), or hash lazily on a background
  thread at boot — not synchronously inside the first page render.
- =log-system-start!=: drop the full =System/getProperties= dump (or log async /
  behind a debug flag).

/Implemented 2026-07-07:/ =views/warm-version-cache!= precomputes the VMS +
app version hashes on a future at startup (called from both entry paths);
=log-system-start!= now logs only java/os/arch + config instead of every
system property.

** DONE 2.4 Verify AOT actually covers the app [S effort to verify, potentially large win]
=packaging/build-uberjar= calls =b/compile-clj= with =:src-dirs ["src"]= while
the project paths are =src/clj= / =src/cljs=. Verify =target/classes= (and the
jar) contain =.class= files for =behave.core=, =behave.server=, all components —
if namespaces are loading from source at runtime, JVM cold start pays full
Clojure compilation. Fix =:src-dirs= if so.

/Verified 2026-07-07:/ =target/behave7.jar= contains ~38k =.class= files
including =behave/core__init.class=, all components and bases — AOT works
(tools.build finds namespaces by reading =ns= forms, not by path).
Direct linking was tried and *reverted*: it breaks datascript
(=Datom.nth= → =nth-datom= invokeStatic signature mismatch on the
forward-declared, primitive-hinted fn) and A/B measurement showed no
class-load win (~0.87 s to =(require 'behave.core)= either way).

** DONE 2.5 JVM-level wins [S effort, medium impact]
- *AppCDS*: JDK 17 supports =-XX:ArchiveClassesAtExit= /
  =-XX:SharedArchiveFile=. Generate an archive during a post-install first run
  (or ship one built in CI) and add the flag to =conveyor.base.conf jvm.options=.
  Typically 20-40% off JVM+Clojure class-load time.

/Investigated 2026-07-07 — JDK bumped, AppCDS blocked by cross-building:/
- *Bumped Conveyor to JDK 21* (Corretto, =/stdlib/jdk/21/amazon.conf=);
  verified a Windows package builds (JAVA_VERSION=21.0.9 runtime).
- *AppCDS benefit confirmed locally*: a static archive over the uberjar cuts
  =(require 'behave.core)= from *0.87s → 0.44s* (~2x on app class loading).
- *But it cannot be enabled in Conveyor packages today*: CDS archives must be
  generated by a JVM executing on the target platform.
  =jlink --generate-cds-archive= fails when cross-linking (tried: Windows
  runtime built on macOS), and CI cross-builds all three platforms from
  Ubuntu. Without a base archive in the jlinked runtime,
  =-XX:+AutoCreateSharedArchive= is a warn-and-ignore no-op (verified).
  Conveyor also strips =lib/classlist=, so a client-side =-Xshare:dump= of
  the base archive is blocked too. Conveyor lists automatic AppCDS as a
  planned future feature.
- *Viable paths if we want the 2x later*:
  1. App-managed training run (Windows only — per-user install dir is
     writable): on first clean exit, spawn the bundled JVM with
     =-XX:DumpLoadedClassList= + =-Xshare:dump -XX:SharedClassListFile= to
     produce a *static* archive next to the JARs, then launch with
     =-XX:SharedArchiveFile=&&/behave7.jsa= (=-Xshare:auto= ignores it until
     it exists). Needs testing on a real Windows install.
  2. Per-platform CI runners so jlink runs natively per target.
  3. Wait for Conveyor's built-in AppCDS support.
- Consider =-XX:TieredStopAtLevel=1= for the desktop app (startup-biased JIT) —
  measure; compute-heavy paths run in WASM on the client, so peak JVM perf
  matters little for the desktop shell. /Skipped for now: would slow the
  background DB restore on large DBs; revisit with measurements./
- =start.sh= places =--add-exports/-Xmx= *after* =-jar=, so they're passed as
  program args, not JVM flags — /fixed 2026-07-07/.
- AppCDS deferred: JDK 17 needs a separate training run
  (=-XX:ArchiveClassesAtExit=); the clean solution
  (=-XX:+AutoCreateSharedArchive=) needs JDK 19+, i.e. a Conveyor JDK bump —
  decide separately.

** DONE 2.6 Fix =/api/init= reconnect churn [S effort, correctness+latency]
=behave.init/init-handler= (=init.clj:39=) does =release-conn!= → reload config
→ reconnect → *another* full =restore-conn= + =build-tx-index!= + full
=export-datoms= on every new-worksheet request. Reuse the existing conn; this
makes "New Worksheet" markedly faster too.

/Implemented 2026-07-07:/ =init-handler= now reconnects only when
=current-worksheet-atom= is set (an =open-handler= call swapped the store to a
temp worksheet file) or the conn is nil; otherwise it reuses the startup
connection.

* Non-Goals / Explicitly Deferred

- Replacing DataScript/SQLite storage layer.
- Shadow-cljs migration (would ease code-splitting but is a separate project).
- Reducing WASM size (454 KB is not on the critical path once 1.1 lands).
